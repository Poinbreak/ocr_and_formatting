Industrial Document Digitization: Progress & Architecture Report

Executive Summary

This report outlines our engineering trajectory in automating the extraction and structuring of complex, handwritten industrial logbooks into machine-readable JSON. We have iteratively evaluated various Vision-Language Models (VLMs) and Optical Character Recognition (OCR) architectures, moving from raw end-to-end model generation to a highly optimized, compute-efficient 3-Stage Pipeline. Our ultimate goal is to achieve 100% pixel-fidelity in reading messy handwriting without language-model hallucinations, followed by strict JSON formatting suitable for database ingestion.

1. Initial Exploration: End-to-End Generative OCR
In our early commits, we attempted to use single, end-to-end multimodal models to both read the image and output structured text.

Step 1: GOT-OCR 2.0
What we tried: Used GOT-OCR-2.0-hf to process images directly.

The Good: Demonstrated that modern models could handle basic layout detection natively.

The Bad: It struggled significantly on CPU-only hardware. Lacking explicit repetition penalties, the model frequently entered infinite generation loops (endlessly repeating numbers or characters). It completely failed on dense, handwritten cursive text.

Step 2: GLM-OCR (0.9B Parameter)
What we tried: Transitioned to zai-org/GLM-OCR, a smaller model optimized for document understanding and handwriting.

The Good: Showed initial promise in maintaining document structure (e.g., grouping lines and utilizing LaTeX syntax for mathematical/tabular data).

The Bad (The Hallucination Problem): We discovered a critical flaw: the model relies too heavily on language priors rather than visual acuity. When faced with messy handwriting, it actively hallucinated words to force sentences to make grammatical sense rather than reading the physical strokes (e.g., misreading "注水西瓜" based on context). It also suffered from repetition loops on CPU.

Verdict: End-to-end small VLMs act as language-model crutches rather than precise OCR engines. They are unfit for industrial applications requiring exact pixel-fidelity.

2. Architectural Pivot: The 3-Stage Pipeline
Realizing that a single model cannot efficiently handle image noise, text extraction, and JSON formatting simultaneously without massive compute (e.g., GPT-4o), we decoupled the architecture into three specialized stages.

Stage 1: OpenCV Image Enhancement
What we tried: Replaced AI-based super-resolution with programmatic Computer Vision (Adaptive Gaussian Thresholding and Dilation).

The Good: Strips out shadows, corrects lighting, and thickens faint pen strokes to produce a high-contrast binary scan. This forces the downstream OCR model to read physical strokes rather than guessing based on blurred artifacts. It operates with virtually zero compute overhead.

Advanced Enhancement Options:

Deskewing (Hough Line Transform): Factory operators often take misaligned photos. OpenCV can detect the baseline angle of the text blocks and rotate the image to be perfectly horizontal before passing it to the VLM.

Illumination Correction via Background Division: Instead of basic thresholding, dividing the image by an estimated background (using heavy Gaussian blur) completely eliminates harsh grease stains or localized shadows that often plague industrial paperwork.

CLAHE (Contrast Limited Adaptive Histogram Equalization): Highly effective for low-light scans where Adaptive Thresholding might wipe out faint pen marks entirely.

Stage 2: Heavyweight VLM OCR Extraction
What we tried: Evaluated larger models (Qwen2.5-VL-3B, PaliGemma-3B, MiniCPM-V 2.6) purely for raw text extraction on the enhanced images. We also evaluated PaddleOCR-VL.

The Good: Qwen2.5-VL proved to be highly capable, demonstrating excellent multi-lingual support and precise document reading. PaddleOCR remains an incredibly robust, production-ready alternative for traditional layout analysis.

The Bad: Downloading and running massive models (like the 16GB MiniCPM) locally on CPU proved highly unstable, leading to frozen downloads and timeouts. By streamlining the pipeline, we bypassed these bottlenecks and zeroed in on Qwen.

Execution Detail: Because Qwen2.5-VL is now isolated strictly to the task of reading (with no prompt instructions to format or interpret), we use a literal, zero-shot prompt. To handle the CPU restrictions effectively, this model should be executed using 4-bit quantization (e.g., via llama.cpp GGUF files) to prevent the freezing and timeouts previously experienced.

Stage 3: NuExtract JSON Formatting
What we tried: Passed the raw, messy OCR text generated by Qwen into numind/NuExtract-tiny.

The Good: NuExtract is a 1.5B parameter model fine-tuned exclusively for template-filling. It successfully mapped the messy, unstructured OCR output into flawless, highly-nested JSON arrays without hallucinating new information. It is incredibly fast and cheap to run locally.

The Bad: In intermediate iterations, we attempted to use NuExtract for text correction, which proved problematic. We subsequently streamlined the pipeline (commit 1cf0dd8) to use NuExtract strictly for schema mapping.

Execution Detail: NuExtract requires a specific prompt formatting mechanism. We supply an empty JSON schema representing the target database structure. The model's singular job is to map the unstructured text block from Qwen into the corresponding string values of the schema. Because it is purely extractive, it effectively eliminates the risk of hallucinating data that was not present in the original log.

3. Current State & Ways to Go From Here
Our current architecture (pipeline_main.py) successfully links OpenCV, Qwen2.5-VL, and NuExtract into a streamlined, high-accuracy pipeline. However, to move this into a production-ready state for the company, we need to address the following requirements:

Immediate Need: Actual Industrial Logs
The Problem: Our current testing has been largely constrained to generalized or limited samples. AI models are highly sensitive to domain-specific formatting (e.g., specific shorthand, specialized tables, or proprietary company forms).

The Solution: We absolutely need a diverse dataset of actual, raw industrial logs used by the company. This will allow us to stress-test the OpenCV enhancement filters against real-world lighting conditions (grease stains, shadows on factory floors) and ensure Qwen can parse the specific handwriting styles of the operators.

Refinement of JSON Formatting
The Problem: While NuExtract works perfectly, it needs a strict, definitive schema to follow.

The Solution: We must collaborate with the database/backend team to define the exact JSON schema required for ingestion. Once defined, we can hardcode this schema into the NuExtract pipeline to ensure every logbook yields standardized, database-ready keys and arrays. Defining this using Pydantic models in Python allows for seamless translation to the required NuExtract template structure.

Future Infrastructure
Running 3B+ parameter models on local CPUs is inefficient for scale. For deployment, we recommend packaging this 3-stage pipeline into a dedicated microservice backed by a lightweight GPU inference engine (like vLLM or Ollama), ensuring instant processing for end-users.
