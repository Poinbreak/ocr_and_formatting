CONFIDENTIAL CASE STUDY

The OCR Dilemma

Modern Vision-Language Models (VLMs) are brilliant reasoners, but terrible transcribers. Their instinct is to translate, summarize, and auto-correct. For ground-truth document extraction, we must identify an engine that suppresses generative habits.

The Zero-Assumptions Protocol

No Auto-Correction

Models are strictly prohibited from fixing grammatical errors or guessing ambiguous cursive. If a stroke is illegible, guessing based on context is a failure condition.

No Unprompted Translation

Multi-lingual extraction must preserve the exact native script without autonomously switching to English.

Spatial Integrity

Linear left-to-right reading is insufficient. Models must prove the ability to parse complex 2D flowcharts, margin notes, and multi-column diagrams correctly.

The 5-Attribute Rubric

Evaluation Attribute Measurement Focus Why it was chosen
Character Accuracy Strict visual-to-text fidelity. Models must not auto-correct technical jargon or hallucinate values.
Geometric Tracking Parsing 2D spatial layouts & tables. Flattening a table into a 1D list destroys critical relational data.
Lexical Segmentation Word and character boundaries. Messy handwriting requires distinct token boundaries without fracturing.
Foundational Strengths Baseline logic (DOMs, Markdown). Indicates underlying architectural maturity and resilience.
Fatal Breakdowns System crashes or infinite loops. A 90% accurate model that occasionally fabricates data is a liability.

Architectural Failures

Cursive Hallucination

Even the top scorer, GLM-OCR, suffered on unstructured cursive (Image 3). Rather than admitting illegibility, its language priors took over, hallucinating completely fabricated entries.

1D Geometric Flattening

Legacy non-generative pipelines like PaddleOCR failed catastrophically. Lacking spatial intelligence, it read across column gaps indiscriminately, flattening 2D tables into vertical, meaningless gibberish strings.

The Strategic Solution:
Qwen 3.5 Series

The data reveals that the highly-efficient Qwen 3.5 models punch massively above their compute class.

The 4B Model: Exhibits brilliant spatial logic—formulating step-by-step row/column extraction plans (Scoring 80 on Image 4)—but truncates early because it wastes token limits "thinking out loud."

The 0.8B Model: The compute optimization king. Highly accurate transcription for a sub-1B parameter size, though it currently flattens 2D structures.

The Fine-Tuning Blueprint

To balance Engineering's mandate for data accuracy with compute reduction, we will execute parameter-efficient fine-tuning (LoRA) on the Qwen 3.5 models.

Execution Plan

1. Hardware Efficiency

Because the 0.8B and 4B models possess micro footprints (1.6GB to 8GB), we can run the entire training pipeline on free Kaggle GPU instances (e.g., single Tesla T4). This reduces R&D cloud training costs to absolutely zero.

2. Dataset Forcing

By training the models exclusively on image-to-JSON pairs, we will rewire their projection layers.

  • 4B Fix: Suppress conversational monologue, curing early truncation.
  • 0.8B Fix: Enforce strict JSON key mapping to cure 2D table flattening.
